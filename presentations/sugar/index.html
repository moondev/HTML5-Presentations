About Sugar

A Quick Summary

Sugar is likely to be compared to Underscore.js as they both are support libraries for "functional" programming. The main differences are in their depth, syntax, and overall implementation. Sugar supports a much larger set of core utilities than that of Underscore.js. Those actively using Underscore.js will likely complain about the fact that Sugar extends Javascript objects. While I cannot argue this fact, Sugar does make an effort to maintain maximum compatibility as it will never overwrite methods that already exist. You can even manually override existing frameworks that have already extended native objects (Prototype, YUI, MooTools) with a simple call to Object.enableSugar().

Components of Sugar

  • Objects
    • Create hash-like objects (keys, values)
    • Type checking methods
  • Arrays
    • Cross-browser fixes for broken methods (indexOf, forEach, reduce...)
    • Enhanced (extended) support of methods (additional arguments)
    • Helper methods for manipulation
  • Strings
    • Cross-browser fixes for regex inconsistencies (split)
    • Multi-lingual support (unicode)
    • Helper methods for manipulation
  • Numbers
    • Shortcut methods to Math object methods
    • Number rounding to precision, number padding, number formatting
    • Intertwined with the date class for advanced date manipulations
  • Dates
    • Understanding of nearly all date strings
      • Text based, relative, absolute, timestamps
    • Date output in nearly any format
    • Helper methods for date manipulation and validation
  • Functions
    • Easily bind scope to functions
    • Delayed function calling
    • Throttled, non-blocking function closures
  • RegExp
    • Fixes inconsistencies in string splitting on regexes
    • Convenient escape method
    • Manipulation methods for generating RegExps

Components: Function Methods

  • Function.lazy(fn, throttle = 1)
    Creates lazy functions for non-blocking operations. This class method will return a function that when called will execute after the CPU is free, preventing "freezing" on heavy operations. Although this will work for a single function, it is best when used in conjunction with loops like each. In such situations, the returned function will be called multiple times. By default each iteration of fn will be called 1ms after the last, or the iteration number * throttle. By passing in a smaller value for throttle (can be a decimal < 1), you can "tighen up" the execution time so that the iterations happen faster. By passing in a greater value for throttle, you can space the function execution out to prevent thread blocking. Playing with this number is the easiest way to strike a balance for heavier operations. example
  • bind(scope, arg1, arg2, ...)
    Binds scope as the this object for the function when it is called. Also allows currying an unlimited number of parameters. example
  • cancel()
    Cancels a function scheduled to be called with delay(). Note: This method won't work when using certain other frameworks like Prototype, as they will retain their delay method. example
  • defer(arg1, arg2, ...)
    Defers the function to be called after the stack is empty (i.e. the CPU becomes idle). Defer can also curry arguments passed in. example
  • delay(duration, arg1, arg2, ...)
    Delays the function for duration milliseconds. Returns the timer to be used with clearTimeout(). In addition to the timer returned, functions using the delay method can be canceled using the cancel() method. Can also curry arguments passed in after the duration. example
